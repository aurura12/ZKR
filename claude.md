# 项目全局开发指南

## 📂 目录结构说明
- `/lab-erp-demo`：包含所有前端 UI 与交互逻辑。
- `/erp-backend`：包含所有后端服务、API 接口与数据库逻辑。
- **核心要求**：开发新功能或修复 Bug 时，必须同时兼顾前后端的联动，确保 API 接口的入参和出参在两端完全一致。

## ⚠️ 核心代码纪律（必须严格遵守）
1. **先思考，后动手**：当处理与代码相关的问题（特别是 bug）时，请先在思考过程中分析问题可能出在哪里，然后主动向我索取相关的目标代码进行详细审阅，不要盲目去整个目录里乱搜乱改。
2. **后台更新优先**：在对代码进行修改后，直接在本地文件中完成更新并简要说明变更结果；**不要在回复中全量输出修改后的完整代码**，除非我明确要求查看完整文件。
3. **最小侵入原则**：尽量避免修改全局性的接口和方法，尽量避免生成新的 entity 和接口。
4. **保持原样**：如果现有的代码逻辑可以通过不修改来解决问题，请绝对不要去触碰或修改它。
5. **数据库绝对安全原则**：在通过 MCP 访问 postgres 数据库时，只能执行 SELECT 读取操作或查看表结构。绝对禁止执行任何 UPDATE、DELETE、DROP、INSERT 等修改数据库状态的 SQL 语句！
6. **禁止临时兼容补丁**：禁止通过容器内临时复制/改名静态资源、手工创建 hash 别名文件、在线热修容器文件等方式修复前端资源问题。必须通过正式代码修复 + 重新构建镜像 + 正式发布解决。

## 🧠 本地记忆缓存纪律（项目级）
1. **每轮先检索记忆**：在答复前优先从本地 memory 检索可复用上下文。
2. **每轮增量写入记忆**：将本轮新增且可复用的信息写入 memory。
3. **冲突新值优先**：新记忆与旧记忆冲突时，先删除旧 observation，再写入新 observation（以新记忆为准）。
4. **仅存可复用事实**：避免写入一次性噪声信息，降低记忆污染。
5. **执行细则**：详见 `.claude/memory_protocol.md`。

## 🎯 财务系统长期宗旨（固定）
- 财务系统必须**真实、准确、及时**映射 ERP 全量业务活动。
- 所有报表与指标必须以 ERP 实际业务数据为准，自动提取与报表相关信息，禁止“脱离业务事实”的展示。
- 任何影响资金、结算、分红、成本、利润的 ERP 事件，都应可追溯地进入财务核算链路。
